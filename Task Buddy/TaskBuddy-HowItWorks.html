Section 2

main.js — The App Entry Point

This is the first file that runs. Its only job is to create the Vue app and plug in the two plugins: Pinia (stores) and Vue Router (navigation). Then it attaches the whole app to the <div id="app"> in index.html.

// src/main.js
import { createApp } from 'vue'
import App          from './App.vue'
import router       from './router'       // our route map
import { createPinia } from 'pinia'       // state manager

const app = createApp(App)   // 1. create the Vue app

app.use(createPinia())        // 2. plug in Pinia (stores)
app.use(router)               // 3. plug in Vue Router

app.mount('#app')           // 4. attach to index.html's <div id="app">
🎯 Key point: app.use(router) is what makes <RouterView /> and <RouterLink /> work everywhere. Without this line, navigation would break entirely.
Section 3

App.vue — The Root Shell

App.vue is the outer container that wraps the entire app. It does two things:

  • Restores the dark/light mode preference from the browser when the app loads
  • Renders <RouterView /> — a slot that Vue Router fills with the correct page component based on the current URL
// src/App.vue
<script setup>
import { onMounted } from 'vue'

onMounted(() => {
  // When app loads, check saved theme preference
  const saved = localStorage.getItem('task-buddy-theme')
  if (saved === 'dark') {
    document.documentElement.classList.add('dark')   // enable dark mode
  }
})
</script>

<template>
  <!-- RouterView fills in with: LoginView, DashboardView, etc -->
  <RouterView />
</template>
💡 Think of it this way: App.vue is like a picture frame. The frame never changes — but the RouterView inside it swaps in different "pictures" (pages) as you navigate.
Section 4

router/index.js — URL Routing

The router is a map of URLs to components. When the user visits a URL, the router finds the matching component and tells <RouterView /> to display it.

// src/router/index.js
import { createRouter, createWebHistory } from 'vue-router'
import { useAuthStore } from '../stores/auth'

// Import each page component
import DashboardView   from '../views/DashboardView.vue'
import LoginView       from '../views/LoginView.vue'
import TodoView        from '../views/TodoView.vue'
import CompletedView   from '../views/CompletedView.vue'

const routes = [
  { path: '/',                   redirect: '/dashboard' },
  { path: '/login',             component: LoginView },
  { path: '/dashboard',         component: DashboardView,
    meta: { requiresAuth: true } },   // ← protected!
  { path: '/dashboard/todo',    component: TodoView,
    meta: { requiresAuth: true } },
  { path: '/dashboard/done',    component: CompletedView,
    meta: { requiresAuth: true } },
]

const router = createRouter({
  history: createWebHistory(),  // clean URLs (no # symbol)
  routes,
})

// 🔒 Route Guard – runs before EVERY navigation
router.beforeEach((to) => {
  const authStore = useAuthStore()

  // If page needs login AND user isn't logged in → redirect to /login
  if (to.meta.requiresAuth && !authStore.isAuthenticated) {
    return { name: 'Login' }
  }
  // Otherwise: let navigation happen
})

export default router

Section 5

stores/auth.js — Login State (Pinia Store)

A Pinia store is a place to keep data that multiple components need to share. The auth store holds who is currently logged in. Any view can read the user's name, check if they're logged in, or call login/logout.

// src/stores/auth.js
import { ref, computed } from 'vue'
import { defineStore }  from 'pinia'

export const useAuthStore = defineStore('auth', () => {

  // STATE – the logged-in user (null if nobody is logged in)
  const user = ref(
    JSON.parse(localStorage.getItem('flowtask_current_user')) || null
  )

  // COMPUTED – true if someone is logged in
  const isAuthenticated = computed(() => !!user.value)

  // ACTION – login: find user in localStorage, return true/false
  function login(email, password) {
    const users = JSON.parse(localStorage.getItem('flowtask_users') || '[]')
    const found = users.find(u => u.email === email && u.password === password)
    if (found) {
      user.value = found
      localStorage.setItem('flowtask_current_user', JSON.stringify(found))
      return true   // success
    }
    return false     // wrong credentials
  }

  // ACTION – logout: clear user data
  function logout() {
    user.value = null
    localStorage.removeItem('flowtask_current_user')
  }

  return { user, isAuthenticated, login, logout }
})
✅ How it connects:
• LoginView.vue calls authStore.login(email, password)
• router/index.js checks authStore.isAuthenticated in the guard
• DashboardView.vue reads authStore.user.username to show the name
• Any view calls authStore.logout() when the user clicks "Sign out"
Section 6

stores/tasks.js — Task Data (Pinia Store)

This store holds the array of all tasks. Every view that shows tasks reads from here. When a task is added, deleted, or its status changed — all views update automatically because they share the same reactive data.

// src/stores/tasks.js
export const useTasksStore = defineStore('tasks', () => {

  // STATE – the task array (pre-loaded with demo tasks)
  const tasks = ref([ ... ])    // array of task objects

  // ACTION – add a brand-new task
  function addTask(task) {
    tasks.value.push({
      id: Date.now(),           // unique ID using timestamp
      title:       task.title,
      description: task.description,
      priority:    task.priority,   // 'low' | 'medium' | 'high'
      dueDate:     task.dueDate,
      category:    task.category,
      status:      'todo',         // always starts as To Do
    })
  }

  // ACTION – change a task's status (todo / inprogress / done)
  function updateStatus(id, newStatus) {
    const task = tasks.value.find(t => t.id === id)
    if (task) task.status = newStatus
  }

  // ACTION – permanently remove a task by its ID
  function deleteTask(id) {
    tasks.value = tasks.value.filter(t => t.id !== id)
  }

  return { tasks, addTask, updateStatus, deleteTask }
})

Section 7

DashboardView.vue — The Main Page

This is the most important view. It connects everything together: it reads from both stores, shows stats, renders the sidebar navigation, lists tasks using the TaskCard component, and opens the CreateTaskModal.

The Script — What it imports and calculates

// src/views/DashboardView.vue — <script setup>

// 1. Import Vue helpers
import { ref, computed }   from 'vue'
import { useRouter, useRoute } from 'vue-router'

// 2. Import both stores
import { useTasksStore } from '../stores/tasks'
import { useAuthStore }  from '../stores/auth'

// 3. Import child components
import TaskCard          from '../components/tasks/TaskCard.vue'
import CreateTaskModal   from '../components/ui/CreateTaskModal.vue'

// 4. Use the stores (this connects to the shared data)
const tasksStore = useTasksStore()
const authStore  = useAuthStore()
const router     = useRouter()
const route      = useRoute()   // tells us the current URL path

// 5. Local state – show/hide the modal
const showModal = ref(false)

// 6. Statistics – auto-calculated from the task list
const today = new Date().toISOString().slice(0, 10)   // "2026-02-22"

const todaysTasks = computed(() =>
  tasksStore.tasks.filter(t => t.dueDate === today).length
)
const completionPercent = computed(() => {
  if (tasksStore.tasks.length === 0) return 0
  const done = tasksStore.tasks.filter(t => t.status === 'done').length
  return Math.round((done / tasksStore.tasks.length) * 100)
})

The Template — What it renders

<!-- src/views/DashboardView.vue — <template> -->
<template>
  <div class="flex min-h-screen">

    <!-- ── SIDEBAR ────────────────────────────── -->
    <aside>
      <span>TaskBuddy</span>

      <nav>
        <!-- Loop over navLinks array to build the nav -->
        <router-link
          v-for="link in navLinks"
          :to="link.to"

        >
          {{ link.label }}
        </router-link>
      </nav>

      <button @click="logout">Sign out</button>
    </aside>

    <!-- ── MAIN AREA ──────────────────────────── -->
    <main>
      <!-- Stats cards -->
      <div>Today: {{ todaysTasks }}</div>
      <div>Completion: {{ completionPercent }}%</div>

      <!-- Create Task button (toggles modal) -->
      <button @click="showModal = true">+ Create Task</button>

      <!-- Task grid – one TaskCard per task -->
      <TaskCard
        v-for="task in tasksStore.tasks"
        :key="task.id"
        :task="task"
        @status-change="handleStatusChange"
      />
    </main>

    <!-- Modal only shows when showModal is true -->
    <CreateTaskModal v-if="showModal" @close="showModal = false" />

  </div>
</template>
🎯 Key concept — v-for: v-for="task in tasksStore.tasks" loops through every task in the store and creates one <TaskCard> for each one. When you add or delete a task from the store, Vue automatically re-renders the list.
Section 8

TaskCard.vue — One Task Card

TaskCard is a reusable component. It receives one task as a prop (input data), displays it, and can emit (send back) events when the user changes the status or clicks delete.

// src/components/tasks/TaskCard.vue
<script setup>

// PROPS – data passed IN from the parent (DashboardView)
const props = defineProps({
  task:       { type: Object,  required: true  },  // the task to show
  showDelete: { type: Boolean, default:  false },  // show delete button?
})

// EMITS – events sent back OUT to the parent
const emit = defineEmits([
  'status-change',   // fired when user changes status dropdown
  'delete',          // fired when user clicks Delete
  'share',           // fired on double-click
])

// Computed – is this task overdue?
const isOverdue = computed(() => {
  const today = new Date()
  today.setHours(0, 0, 0, 0)
  return new Date(props.task.dueDate) < today && props.task.status !== 'done'
})

// When status dropdown changes, tell the parent
function onStatusChange(event) {
  emit('status-change', { id: props.task.id, status: event.target.value })
}
</script>

<!-- Template shows: priority badge, title, description, due date,
     status dropdown, and optional delete button -->

Props vs Emits — the two-way conversation

DashboardView
──── :task="task" ────▶
TaskCard
DashboardView
◀── @status-change ────
TaskCard
💡 Rule: Data flows down via props. Events flow up via emits. The parent (Dashboard) is always in control of the actual data — TaskCard just displays it and reports user actions.
Section 9

CreateTaskModal.vue — The "New Task" Popup

This component appears as an overlay popup when you click "+ Create Task". It collects task details from the user, then calls tasksStore.addTask() directly to save the task.

// src/components/ui/CreateTaskModal.vue
<script setup>
import { useTasksStore } from '../../stores/tasks'

const emit       = defineEmits(['close'])   // tells parent to hide modal
const tasksStore = useTasksStore()

// Local state for the form fields
const title    = ref('')
const priority = ref('medium')
const dueDate  = ref('')
const category = ref('')

function handleCreate() {
  if (!title.value.trim()) return   // don't save empty tasks

  // Save the task directly to the Pinia store
  tasksStore.addTask({
    title:    title.value,
    priority: priority.value,
    dueDate:  dueDate.value,
    category: category.value,
  })

  emit('close')   // tell DashboardView to hide the modal
}
</script>
✅ Flow: User fills form → clicks "Create Task" → handleCreate() runs → calls tasksStore.addTask() → new task appears in store → DashboardView's v-for loop automatically renders the new card → modal closes.
Section 10

Complete Request Flow — Start to Finish

Here are the most common user journeys and exactly what happens in the code at each step.

🔑 Flow 1: User Logs In

1
User visits / or /dashboard Router's beforeEach guard fires. Checks authStore.isAuthenticated → it's false → redirects to /login.
2
LoginView.vue renders User types email + password into the form and presses "Sign In" → handleLogin() runs.
3
authStore.login(email, password) is called Looks up the user in localStorage. If found → sets user.value and saves to localStorage → returns true.
4
router.push('/dashboard') is called Guard checks again → isAuthenticated is now true → allows navigation → DashboardView renders.

➕ Flow 2: User Creates a Task

1
User clicks "+ Create Task" in DashboardView showModal = true → v-if="showModal" makes <CreateTaskModal> appear.
2
User fills form, clicks "Create Task" handleCreate() in CreateTaskModal runs → calls tasksStore.addTask({...}).
3
Store adds the task tasks.value.push({id: Date.now(), status: 'todo', ...}). Since tasks is reactive, every component watching it updates instantly.
4
Modal emits 'close' emit('close') → parent DashboardView sets showModal = false → modal disappears. New task card is already visible.

🗑️ Flow 3: User Deletes from Completed

1
User clicks "Delete" in CompletedView TaskCard emits 'delete' with the task's id → CompletedView's handleDelete(id) handler fires.
2
tasksStore.deleteTask(id) is called tasks.value = tasks.value.filter(t => t.id !== id) — removes the task from the shared array.
3
All views update automatically Because all views read from the same tasksStore.tasks reactive array, the task disappears from Dashboard, Completed, and everywhere else instantly — no extra code needed.
